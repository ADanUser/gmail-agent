# Before / After Report — LLM Provider Switch

## Контекст

Gmail Support Agent изначально использовал бесплатные модели через OpenRouter
(`openrouter/free` — auto-роутинг между разными free моделями). При прогоне
eval harness (5 сценариев, golden dataset) обнаружились проблемы со
стабильностью и скоростью. Решение: переключиться на Groq (`llama-3.3-70b-versatile`).

## Метод измерения

Eval harness (`eval_runner.py` + `eval_harness.py`) — 5 тестовых сценариев,
каждый сценарий с известным ожидаемым поведением агента (вызвать `send_email`
с правильным смыслом ответа, или корректно НЕ отвечать). Проверка через
LLM-as-judge для смысловой точности вместо точного совпадения слов.

## Результаты

| Метрика | Before (OpenRouter free) | After (Groq) |
|---|---|---|
| Pass rate | 3-4 из 5 (60-80%), нестабильно между прогонами | 5 из 5 (100%) |
| Latency на сценарий | 4.3 – 32.6 сек | 0.4 – 2.5 сек |
| Средняя latency | ~12 сек | ~1 сек |
| Стабильность вызова tool | Иногда `None` вместо tool call или текста | Стабильно вызывает нужный tool |
| Ошибки Rate Limit | Дневной лимит исчерпан после ~10-15 запросов | Не встречалось при том же объёме тестов |

## Конкретные провалы на OpenRouter free (зафиксировано в eval_report.json)

- **eval_001 (возврат)** — в одном из прогонов агент вообще не вызвал `send_email`
- **eval_002 (статус заказа)** — формулировка ответа не совпадала с ожидаемым словом при строгой проверке (что и привело к переходу на LLM-as-judge вместо exact match)
- Несколько прогонов завершились `RateLimitError: 429` после исчерпания дневного лимита free tier (50 запросов/день)

## Известное ограничение после перехода на Groq

В одном из ответов (eval_002) встретился артефакт смешения языков —
слово `đang` (вьетнамское) внутри русского текста письма. Смысл ответа не
пострадал, eval прошёл успешно. Рассмотрены варианты: (1) добавить
guardrail на санитизацию языка перед отправкой, (2) оставить как есть.
Принято решение не исправлять для текущей версии — риск признан низким
(единичное наблюдение, не влияет на смысл письма), а добавление ещё одного
guardrail ради редкого артефакта генерации не оправдано на этом этапе
проекта. Если частота повторится на более широком eval-наборе — вернуться
к варианту (1).

## Вывод

Для production-агента, который должен работать стабильно и быстро, free tier
OpenRouter не подходит — слишком нестабилен в вызове tools и имеет жёсткий
дневной лимит запросов. Groq показал кратное улучшение по всем метрикам при
сохранении нулевой стоимости (free tier с щедрым лимитом). Рекомендация:
использовать Groq как основной провайдер для разработки и демо, с пониманием
что для реального продакшена с высоким SLA нужна платная модель
(Anthropic/OpenAI) с дополнительным мониторингом качества ответов.

## Связанные документы

Отдельная от производительности провайдера логическая проблема — агент
вызывает `send_email` даже когда условие задачи не выполнено — описана и
