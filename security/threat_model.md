# Threat Model — Gmail Support Agent

## Что защищаем (Assets)

| Актив | Описание | Ценность |
|---|---|---|
| Gmail аккаунт | Доступ к реальной почте пользователя | Критическая |
| Содержимое писем | Личные данные клиентов, переписка | Высокая |
| Возможность отправки | Агент может отправлять письма от имени пользователя | Критическая |
| OAuth токен | `token.pickle` даёт доступ к Gmail без пароля | Критическая |
| API ключи | `GROQ_API_KEY` (активный, используется провайдером Groq); `OPENROUTER_API_KEY` (legacy — код клиента оставлен закомментирован в `agent.py` как fallback-референс, в текущей конфигурации не используется) | Высокая |

---

## Кто может атаковать (Attackers)

| Злоумышленник | Мотив | Метод |
|---|---|---|
| Внешний злоумышленник | Рассылка спама, кража данных | Prompt injection через входящие письма |
| Недобросовестный клиент | Получить чужие данные, манипулировать агентом | Социальная инженерия в письме |
| Скомпрометированный сервер | Вредоносный контент в данных | Indirect injection через ответы API |
| Сама модель (непреднамеренно) | Нет злого умысла, но модель может действовать некорректно из-за ограничений LLM | Hallucinated tool call — вызов действия без основания в данных |

## Сценарии атак и защиты (Abuse Cases)

| ID | Атака | Вектор | Статус | Защита |
|---|---|---|---|---|
| T-001 | Prompt injection через тело письма | Indirect injection | ✅ Закрыто | `detect_injection_in_email()` блокирует на входе |
| T-002 | Подмена адреса получателя | Tool misuse | ✅ Закрыто | `validate_send_email()` сверяет с отправителем |
| T-003 | Социальная инженерия (фейк-администратор) | Identity abuse | ✅ Закрыто | Injection patterns + hardened prompt |
| T-004 | Спам через агента | Excessive agency | ✅ Закрыто | Blocklist доменов в `guardrails.py` |
| T-005 | Вредоносные ссылки в теле ответа | Data injection | ✅ Закрыто | URL pattern detection в `guardrails.py` |
| T-006 | Утечка токена OAuth | Credential theft | ⚠️ Частично | `.gitignore`/`.dockerignore` защищают от случайного коммита или встраивания в образ, но не от кражи с диска |
| T-007 | Инъекция через вложения (PDF, DOCX) | Malicious docs | ⚠️ Частично | Агент не читает вложения (не реализовано), риск при расширении |
| T-008 | Перебор API ключей | Brute force | ✅ Закрыто | `RateLimiter` в `guardrails.py` — не более 10 писем в минуту |
| T-009 | Вызов tool при невыполненном условии задачи (агент "галлюцинирует" действие) | Excessive agency / hallucinated action | ✅ Закрыто | Найдено на практике: агент искал письмо от конкретного отправителя, не находил, но всё равно вызывал `send_email` с фиктивным или неверным адресом. Закрыто двумя независимыми guardrail'ами в `agent.py`: (1) блокировка `send_email` без предварительного `get_email_by_id`, (2) `validate_send_email()` сверяет адрес получателя с реальным отправителем открытого письма |

## Принципы защиты применённые в проекте

**Least Privilege**
- Gmail scopes: только `readonly` и `send` — агент не может удалять письма или менять настройки
- Агент не имеет доступа к базам данных, файловой системе или другим сервисам

**Defence in Depth (несколько слоёв)**
1. Проверка входящего письма на injection (`detect_injection_in_email`)
2. Hardened system prompt с явными правилами безопасности
3. Валидация каждого tool call перед выполнением (`validate_send_email`)
4. Проверка предпосылок действия — `send_email` невозможен без предварительного `get_email_by_id` (T-009)
5. Dry-run / approve режим для необратимых действий

**Auditability**
- Каждое действие агента логируется в `audit_log.jsonl`, включая заблокированные попытки (статус `blocked`)
- Каждый прогон сохраняется в `traces/` с полным разбором шагов

## Известные ограничения

1. **System prompt не достаточен как единственная защита.** Подтверждено дважды независимо: attack_001 прошёл через hardened prompt в Block 10 (потребовался code guardrail), и T-009 — модель вызывала `send_email` с фиктивным адресом даже после явного запрета в промпте "не вызывай tool если условие не выполнено". Вывод: prompt engineering снижает частоту нежелательного поведения, но не исключает его — критичные действия должны проверяться независимым кодом, а не только инструкцией модели.

2. **OAuth токен истекает** — в Testing режиме Google токен живёт 7 дней. Для продакшена нужна верификация приложения.

3. **Вложения не обрабатываются** — если в будущем агент будет читать PDF/DOCX вложения, нужна отдельная защита от malicious docs injection.
