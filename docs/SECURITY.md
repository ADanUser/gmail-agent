# Security Checklist — Gmail Support Agent

## Scopes (минимальные права доступа)

| Scope | Зачем нужен | Риск если скомпрометирован |
|---|---|---|
| `gmail.readonly` | Агент читает входящие письма для анализа запросов клиентов | Утечка содержимого писем |
| `gmail.send` | Агент отправляет ответы клиентам | Отправка нежелательных писем от имени пользователя |

**Что НЕ запрошено намеренно:**
- `gmail.modify` — агенту не нужно удалять или менять письма
- `https://mail.google.com/` (полный доступ) — слишком широкий scope, не нужен для support-задач

## Secrets — где хранятся

| Секрет | Где хранится | Как защищён |
|---|---|---|
| `GROQ_API_KEY` | `.env` файл | Не коммитится в Git (`.gitignore`), не копируется в Docker-образ (`.dockerignore`) |
| `credentials.json` (OAuth client) | Локально в папке проекта | Не коммитится в Git, не копируется в Docker-образ |
| `token.pickle` (refresh/access token) | Локально в папке проекта | Не коммитится в Git, не копируется в Docker-образ, токен истекает через 7 дней (Testing mode) |

Примечание: изначально использовался `OPENROUTER_API_KEY` (OpenRouter),
но из-за нестабильности и лимитов бесплатного тарифа проект переключён на
Groq. Код клиента под OpenRouter оставлен закомментированным в `agent.py`
как fallback-референс, но переменная `OPENROUTER_API_KEY` в текущей
конфигурации не используется.

В Docker-контейнер секреты передаются снаружи при запуске
(`--env-file .env` и `-v token.pickle:/app/token.pickle`), а не
встраиваются в образ — см. `docs/RUNBOOK.md`.

## Approval policy — какие действия требуют подтверждения

| Действие | Требует approval? | Почему |
|---|---|---|
| `read_emails` | Нет | Безопасное чтение, не меняет состояние |
| `get_email_by_id` | Нет | Безопасное чтение |
| `send_email` | **Да (dry-run)** | Необратимое действие — письмо уходит реальному человеку |

## Известные риски и ограничения

1. **OAuth в режиме Testing** — токен истекает каждые 7 дней, требует
   повторной авторизации. Для продакшена нужна верификация приложения в
   Google.

2. **LLM может вызвать tool даже когда условие задачи не выполнено.**
   Наблюдалось на практике: агент искал письмо от определённого отправителя,
   не находил его, но всё равно вызывал `send_email` с фиктивным или
   невалидным адресом получателя вместо того, чтобы сообщить об отсутствии
   условия текстом. Промпт-хардненинг (явное указание "если условие не
   выполнено — не вызывай tool") снижает частоту, но не устраняет
   полностью — модель может обойти инструкцию. Из-за этого агент **нельзя**
   запускать в полностью автономном режиме (`dry_run=False`) без
   независимых guardrails на уровне кода и человека в цикле.

   Закрыто двумя независимыми проверками в `agent.py`/`guardrails.py`:
   - Блокировка `send_email`, если перед этим не было вызова
     `get_email_by_id` (агент не может ответить на письмо, которое не
     открывал).
   - `validate_send_email()` — сверяет адрес получателя с реальным
     отправителем открытого письма и блокирует несовпадение.

3. **Prompt injection через письма — протестировано в Block 10.**
   `guardrails.py` содержит `INJECTION_PATTERNS` и функцию
   `detect_injection_in_email()`, которая проверяет входящие письма на
   попытки вида "игнорируй прошлые инструкции" до передачи их агенту.
   `security_tests.py` покрывает 5 типов атак, включая prompt injection
   через тело письма и попытку подмены адреса получателя. Полный отчёт —
   в `security/threat_model.md`.

4. **Audit log не защищён от изменения** — `audit_log.jsonl` хранится
   локально без подписи/хеширования. Для продакшена нужен immutable log
   (например, запись в отдельную БД с правами только на добавление).

## Что сделано правильно

- Минимальные scopes (least privilege)
- Все секреты вне Git и вне Docker-образа, передаются во время выполнения
- `send_email` защищён approval-шагом и двумя независимыми guardrails
  на уровне кода (не полагается только на поведение модели)
- Все действия логируются в audit log с timestamp, параметрами и
  результатом, включая статус `blocked` для заблокированных guardrail'ами
  попыток
- Rate limiting на `send_email` (не более 10 писем в минуту) защищает от
  runaway agent и потенциального спама
